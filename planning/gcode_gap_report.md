# G-Code Gap Report — rs_cam vs Fusion reference

**Phase 0 deliverable.** Per-dialect, per-fixture summary of how the **current** rs_cam emitter (`crates/rs_cam_core/src/gcode.rs`) differs from what Fusion's published `.cps` posts would produce for an equivalent toolpath.

**Status (post-Phase 4b):** spec-only gaps captured (Phase 0); emulator validation suite in place (Phase 0.5 + 4a); validator baseline locked in (Phase 1); corpus broadened from 6 to 16 fixtures (Phase 4b). The "rs_cam current" column is filled from the captures in `planning/gcode_current_outputs/` (now 64 captures across 16 fixtures × 4 dialects, including grblHAL). 96 emulator-validation tests, all green.

**Original target — Fusion byte-parity — was abandoned because Autodesk's `post` CLI doesn't exist for Linux** (it ships only inside Fusion 360, which has no native Linux build; cam-posteditor's binary lookup hardcodes Win/Mac install paths). Re-targeted to controller-emulator validation — which tests what actually matters (does the controller accept this g-code?) and is hermetic on Linux for CI use.

**Sources:**
- rs_cam current outputs: `planning/gcode_current_outputs/<fixture>_<dialect>.nc` (committed)
- Fusion `.cps` source: `reference/posts/{grbl,linuxcnc,mach3mill}.cps` (gitignored — see [Phase 0 license decision](GCODE_EXPORT_OVERHAUL.md#phase-0--reference-corpus--gap-report))
- Reference notes: `planning/post_reference_notes.md`
- Fixture corpus: `planning/fixtures/README.md` and `crates/rs_cam_core/tests/gcode_phase0_capture.rs`

## TL;DR — gaps surfaced from the `.cps` spec alone

These are issues we can identify *without* needing Fusion's actual output, by comparing rs_cam captures against what the `.cps` source code says Fusion will emit. They are graded by safety impact:

### 🔴 Machine-safety class (must fix in Phase 4 at the latest)

1. **LinuxCNC: missing `G91.1` in preamble.** Fusion's linuxcnc.cps emits `G90 G94 G17 G91.1` to put arcs in incremental-IJK mode. **rs_cam emits no `G91.1`**, which means `G2/G3 I.. J..` blocks may be interpreted as absolute IJK on a fresh LinuxCNC install. Look at `f2_arcs_xy_linuxcnc.nc` line 7: `G2 X0.0000 Y10.0000 Z-2.0000 I-10.0000 J0.0000` — without G91.1, LinuxCNC reads `I-10` as the absolute X coordinate of the arc center, not as `start.x - 10`. **This is a latent machine-crash bug** for any user who hasn't manually configured their `[EMC]` ini to default `G91.1`.

2. **LinuxCNC: emits `M2` instead of `M30`.** Fusion uses `M30` (program end with rewind/reset). rs_cam emits `M2` (program end without reset), which leaves modal state pinned across program restarts on some controllers. See `f1_basic_lines_linuxcnc.nc` last line.

3. **LinuxCNC: missing `%` wrapping.** Fusion brackets the program with `%` at the very first and very last lines. rs_cam omits both. Without leading `%`, some LinuxCNC streamers refuse the program; without trailing `%`, the controller may continue parsing into garbage memory.

### 🟡 Encoding-correctness class (must fix for golden parity)

4. **Feed decimals reversed for Grbl vs Mach3.** Fusion emits Grbl with 1 decimal (`F600.0`) and Mach3 with 0 decimals (`F600`). rs_cam does the opposite: Grbl `F600` (0 decimals), Mach3 `F600.0` (1 decimal). See `f1_basic_lines_grbl.nc` line 5 (`F600`) vs `f1_basic_lines_mach3.nc` line 6 (`F600.0`).

5. **End-of-program retract is per-dialect ad hoc.** rs_cam emits:
   - Grbl: `G0 Z10.000` (literal hard-coded value, ignores actual safe-Z)
   - LinuxCNC: `G53 G0 Z0` (correct; matches Fusion)
   - Mach3: `G0 Z10.000` followed by `G28 G91 Z0` (mixed/redundant — Fusion does only `G53 Z0` or `G28 Z0`, not both)

6. **`G54` only emitted by LinuxCNC.** Fusion emits the WCS code (`G54..G59`) on every dialect when the toolpath enters its first section. rs_cam emits it only for LinuxCNC. Grbl in particular needs explicit `G54` because some Grbl forks default to `G53` (machine coordinates) on power-on.

7. **Spindle warmup `G4 P2` only on Mach3.** rs_cam's Mach3 post inserts a 2-second dwell after `M3 S<rpm>`. Fusion does NOT do this on any of the three dialects. (It's a sensible safety habit, but it should be a configurable knob, not a hard-coded per-dialect choice.)

8. **`G40 G49 G80` in preamble is overzealous.** rs_cam emits all three cancel-codes on every dialect. Fusion only emits `G40 G49` (and G80 when relevant) on Mach3 / LinuxCNC; Grbl gets a minimal preamble (`G90 G94`, then `G17`, then units). See `f1_basic_lines_grbl.nc` line 2 vs the Grbl `.cps` `writeProgramStart` (lines 255-261).

### 🟢 Cosmetic / metadata class (nice-to-have)

9. **Header comments are sparse.** rs_cam: `(Generated by rs_cam)`. Fusion: program name, program comment, `(Machine: vendor / model)`, then a tool list with each tool's diameter, corner radius, and Z-range. Useful for operators staring at the file before running it.

10. **No blank-line separation between sections.** Fusion writes a `\n` between operations, then the operation comment, then the section content. rs_cam runs everything together (only F5 / F6 emit operation comments at all).

11. **Comment case.** Fusion's LinuxCNC post forces upper-case for comments (`(GENERATED BY RS_CAM)`); rs_cam preserves case. Strictly cosmetic.

## Phase 4b broadened-corpus findings (F7-F16) — **all four bugs FIXED**

10 new fixtures added covering full-circle, X-only feed, ramp-into-arc, sub-mm arcs, depth-step boundaries, tool-change-at-Z-zero, climb-vs-conventional, multi-line pause messages, embedded-newline pre/post snippets, and G41/G40 cutter-comp round-trip. The corpus surfaced four real emitter bugs; **all four are now fixed in the emitter** and the corresponding emulator-validation tests have flipped from `assert_*_rejects_any` to `assert_*_accepts`.

### ✅ Bug 1 — F10 sub-mm arcs (FIXED)

**Was:** gvalidate exit 33 ("arc trajectory inconsistent"), rs274ngc exit 1. Affected all four dialects.

**Fix:** Every shipped post sets `arc_linearize.enabled = true` (threshold 0.05mm). The emitter computes the arc radius from incremental IJK and substitutes a `G1` chord when the radius is below the threshold. F10 captures now contain straight chords; every parser accepts.

### ✅ Bug 2 — F14 multi-line pause messages (FIXED)

**Was:** `render_comment` wrapped messages in `(...)` without sanitizing embedded `\n`; the second line emerged as bare g-code, parser rejected.

**Fix:** `render_comment` and `render_program_pause` now collapse `\n` → ` / `, `\r`/`\t` → single space. The rendered comment stays on one line. F14 captures now show e.g. `(Setup change: Bottom / Flip stock 180 then resume)` — single block, parser-safe.

### ✅ Bug 3 — F15 user pre/post snippets bypass post compatibility (FIXED)

**Was:** Grbl post accepted user `pre_gcode = "M7"` unchanged; Grbl 1.1 rejects M7.

**Fix:** New `PostDefinition.unsupported_mcodes: Vec<u32>` field. Grbl post lists `[7]`. Emitter scans every `Statement::Raw` line; lines containing a denylisted M-code are dropped and replaced with a warning comment naming the dropped command (`(WARNING: M7 unsupported on GRBL; dropped: M7)`). Grbl now accepts the F15 capture.

### ✅ Bug 4 — F16 cutter compensation emitted regardless of post support (FIXED for Grbl-family)

**Was:** `controller_compensation = Some(Left)` always emitted `G41 D<n>` + `G40` regardless of post support. Grbl 1.1 has no comp.

**Fix:** New `PostDefinition.supports_cutter_comp: bool` field (default `true`; Grbl/grblHAL set `false`). Emitter drops G40/G41/G42 lines from `Statement::Raw` and replaces each with a warning comment when the post doesn't support comp. Grbl + grblHAL F16 captures now run cleanly.

LinuxCNC/Mach3 still emit comp codes (they support it); their `rs274ngc` test continues to reject because rs274's offline parse can't validate kerf-shifted geometry without a runtime contour. **This is a validator limitation, not an emitter bug** — the emitted g-code is correct for those controllers.

### Status

All four bugs closed. 96 emulator-validation tests green. The Phase 5 wizard inherits a backend where the only documented "expected-rejects" are honest validator-proxy limitations (gvalidate parsing M7/M6/G41 from the wrong dialect; rs274ngc needing runtime contour for comp), not emitter bugs.

### 🔴 F5 Grbl — REAL BUG: rs_cam emits `M6` for Grbl tool changes

gvalidate exits with **error 20** ("Unsupported or invalid g-code command found in block") on `f5_two_tool_changes_grbl.nc`, line containing `M6 T2`. This is a confirmed bug:

- **Grbl 1.1 does not implement M6.** It's an automatic tool changer command; Grbl is a hobbyist controller without ATC support.
- Fusion's grbl.cps has `useM06` as a property defaulting to `false` for exactly this reason — they use a `T<n>` line plus a `(CHANGE TO T<n>)` operator-prompt comment instead.
- rs_cam's current Grbl post emits `M6 T2` unconditionally. **Any Grbl user with multi-tool g-code from rs_cam right now gets a parse error before the second toolpath starts.**

**Fix path:** Phase 3 when posts become data-driven TOML — add a `tool_change.use_m6` field to `PostDefinition`, default `false` for the Grbl post. Until then, document this as a known issue in the user-facing wizard.

### 🟡 F5 LinuxCNC / Mach3 — PROXY LIMITATION (not bugs)

`gvalidate` is the Grbl parser. M6 IS valid in both LinuxCNC and Mach3, but `gvalidate` doesn't implement it, so it rejects these outputs too. **The output is correct for those controllers — gvalidate just can't validate them.**

**Resolution path:** Phase 4 — install a real LinuxCNC `rs274ngc` parser (build from source on Ubuntu 24.04, ~30 min plus boost/glib deps). At that point flip these tests from `assert_gvalidate_rejects` to `assert_rs274_accepts`.

grblHAL fixtures defer to Phase 4 (no rs_cam grblHAL post yet, and `grblHAL_validator` has a headless-init hang to debug).

## Emulator-validation workflow

**Validators (all installed under `reference/validators/`, gitignored):**

| Dialect | Tool | Source | Build |
|---------|------|--------|-------|
| Grbl | `gvalidate` | [grbl/grbl-sim](https://github.com/grbl/grbl-sim) | `make` after `git clone`; produces `gvalidate` binary |
| grblHAL | `grblHAL_validator` | [grblHAL/Simulator](https://github.com/grblHAL/Simulator) | `make` in `validator/` subdir |
| LinuxCNC + Mach3 (proxy) | LinuxCNC `rs274ngc` parser | [LinuxCNC/linuxcnc](https://github.com/LinuxCNC/linuxcnc) `src/emc/rs274ngc/` | `apt install linuxcnc-uspace` (preferred) or build from source |

**Per-capture run:** for each `planning/gcode_current_outputs/<fixture>_<dialect>.nc`:
- Pipe to the dialect's validator on stdin or pass as argument.
- Capture exit code + stderr.
- Exit 0 → ✅ pass; non-zero → ❌ fail with diagnostic excerpt.

**Test harness:** `crates/rs_cam_core/tests/gcode_emulator_validation.rs` — one `#[test]` per (fixture × dialect), each spawns the validator, asserts exit 0 (or expected non-zero with documented reason). Skip with a clear message if the validator binary is missing (so the test gate degrades gracefully when contributors haven't built the validators).

## Original recommendation for Phase 1 entry (now closed)

Phase 0's stated exit was "we know exactly how far we are." Combining the `.cps`-source-only review with Phase 0.5 emulator results, we now have:

**Confirmed bugs (real, validator-caught):**
1. 🔴 **Grbl tool-change emits unsupported `M6`** — confirmed by gvalidate exit 20 on F5. Multi-tool Grbl programs from rs_cam fail before the second toolpath. Fix: add `use_m6` toggle to `PostDefinition`, default false for Grbl post.

**Likely bugs (from `.cps`-spec review, not yet emulator-confirmed pending Phase 4 LinuxCNC parser):**

2. 🔴 **LinuxCNC missing `G91.1` in preamble** — without it, arcs may be misinterpreted as absolute IJK on a fresh LinuxCNC install. Latent crash risk.
3. 🔴 **LinuxCNC emits `M2` instead of `M30`** — modal state not reset across program restarts.
4. 🔴 **LinuxCNC missing `%` program brackets** — some streamers refuse the program; controller may parse into garbage memory.

**Encoding inconsistencies (no immediate safety risk):**

5. 🟡 Feed decimals reversed for Grbl vs Mach3 vs Fusion convention (we do 0/1, Fusion does 1/0).
6. 🟡 End-of-program retract is per-dialect ad hoc (Grbl hardcodes Z=10 ignoring safe-Z; Mach3 does both `G0 Z` and `G28 Z` redundantly).
7. 🟡 `G54` only emitted by LinuxCNC; Grbl/Mach3 silently rely on machine default.
8. 🟡 Spindle warmup `G4 P2` hardcoded for Mach3 only — should be a per-post knob.
9. 🟡 `G40 G49 G80` over-emitted in Grbl preamble.

**Phase 1 validator scope** — encode invariants that catch each of the above:

- `M6` only allowed when post declares `use_m6 = true` (catches bug 1 once Phase 3 lands)
- `G91.1` required in LinuxCNC preamble (catches bug 2)
- Program-end M-code matches post's declared end code; LinuxCNC must use `M30`, not `M2` (catches bug 3)
- LinuxCNC programs must be `%`-bracketed (catches bug 4)
- Feed decimals must match post's declared format (catches bug 5)
- WCS (G54..G59) emitted at first section on every dialect (catches bug 7)
- Spindle warmup dwell required only when post declares `warmup_ms > 0` (catches bug 8)

The validator runs on `gcode_current_outputs/` snapshots and produces a per-fixture findings count. Goal: drive that count to zero through a series of small, well-tested fixes in Phase 2/3 (IR + data-driven post). The emulator validation suite is the safety net underneath: any commit that breaks Grbl/LinuxCNC/Mach3 parser acceptance gets caught by `cargo test --test gcode_emulator_validation`.
